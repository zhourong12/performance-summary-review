# 自我总结审核标准

**仅审核 `personalSummary`（自我总结 / 评分详情汇总）**，不审核各指标分项 comment、不对照 indicators 做相关性判断。

## 1. 充实度

### 字数

| 级别 | 条件 | 处理 |
|------|------|------|
| 空 | 空白或仅空格 | 不通过 |
| 极短 | < 20 字 | 不通过 |
| 过短 | 20–49 字 | 建议修改 |
| 合格 | ≥ 50 字 | 继续套话检查 |

### 套话 / 空洞表述

以下模式若占 personalSummary **主体**（>60%），判为「空洞」：

- 「完成了本职工作」「表现良好」「整体表现良好」「继续努力」
- 「按时完成各项任务」「积极配合团队」「工作认真」
- 纯形容词堆砌，无具体事项、数据、项目或结果

| 情况 | 处理 |
|------|------|
| 合格字数但主体为套话 | 建议修改；极短套话（如仅「表现良好」6 字）→ 不通过 |
| 合格字数且有具体工作描述 | 通过 |

## 2. 不审核项（明确排除）

以下**不得**作为通过/不通过、驳回的依据：

- `selfReview[].comment` 是否为空（各指标评分详情）
- personalSummary 是否提及某指标名称
- personalSummary 与 `indicators` 工作内容是否「相关」
- 自评分数高低

## 3. 示例

### 通过

**personalSummary**（68 字）：
> 本月完成 A 项目核心模块开发，按时交付 2 个迭代；参与 2 次代码评审并修复 3 个严重缺陷；协助新人完成 onboarding。

### 建议修改

**personalSummary**（32 字）：
> 本月工作认真，完成了领导交办的任务，团队协作良好。

**问题**：过短、套话、无具体事项。

### 不通过

**personalSummary**：空白、`无`、或仅「表现良好」「整体表现良好」等极短套话。

## 4. 批量审核优先级

1. 不通过（空 / 极短套话）
2. 建议修改（过短或空洞）
3. 通过

## 5. 驳回判定

| 审核结论 | 默认动作 |
|----------|----------|
| 不通过 | **建议驳回**；驳回原因只写 personalSummary 问题 |
| 建议修改 | 默认仅报告；用户明确要求时再驳回 |
| 通过 | 不可驳回 |

### 常见驳回场景

- personalSummary 为空
- 极短且无实质内容（如「整体表现良好」）
- 套话占比过高，无任何具体工作描述

### 不宜驳回

- personalSummary ≥ 50 字且含具体工作/成果描述
- 用户仅要求「看看总结」未授权驳回
- 记录仍在 `self_review`（员工未提交）
- 当前账号不满足 integration 中的驳回角色

### 驳回原因勿写

- 「各指标评分详情为空」
- 「总结与某某指标无关 / 未覆盖指标」
